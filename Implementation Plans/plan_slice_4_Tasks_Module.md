# План реализации Среза №4: Задачник и Календарь (Tasks and Calendar Module)

Этот срез посвящен реализации Модуля 4 «Задачник / Планировщик» согласно `docs/business_logic.md` с интеграцией ключевых дат из Модуля Заказов.

## Шаги реализации

### 1. Слой БД (`backend/models.py`)
- **Добавление перечисления приоритетов (Enum)**:
  - Создать `TaskPriority` со значениями: `низкий`, `средний`, `высокий`, `критический`.
- **Добавление перечисления статусов задач (Enum)**:
  - Создать `TaskStatus` со значениями: `нужно сделать`, `в процессе`, `выполнено`, `отменено`.
- **Создание модели `Task`**:
  - `id` (Integer, Primary Key, autoincrement).
  - `title` (String, NOT NULL) — название задачи.
  - `description` (Text, nullable) — описание задачи.
  - `due_date` (Date, nullable) — крайний срок (дедлайн).
  - `priority` (Enum `TaskPriority`, NOT NULL) — приоритет задачи.
  - `status` (Enum `TaskStatus`, NOT NULL, по умолчанию `нужно сделать`) — статус выполнения.
  - `assigned_to` (String, NOT NULL, по умолчанию `DIRECTOR`) — исполнитель.
  - `order_id` (Integer, ForeignKey к `orders.id`, nullable) — связь с заказом.
  - `created_at` (DateTime, default=datetime.now(timezone.utc)) — дата создания.
- **Связи (Relationships)**:
  - В модели `Order`: `tasks = relationship("Task", back_populates="order", cascade="all, delete-orphan")`
  - В модели `Task`: `order = relationship("Order", back_populates="tasks")`
  - В модели `Task`: Добавить `@property def order_product_type(self)` для удобства отображения названия изделия в UI.

### 2. Схемы API (`backend/schemas.py`)
- **`TaskBase`**: содержит общие поля `title`, `description`, `due_date`, `priority`, `status`, `assigned_to`, `order_id`.
- **`TaskCreate`**: наследуется от `TaskBase` (поля `priority` и `status` имеют стандартные дефолты в соответствии с ТЗ/БД).
- **`TaskStatusUpdate`**: содержит только `status` (тип `TaskStatus`).
- **`TaskResponse`**: наследуется от `TaskBase`, содержит `id`, `created_at`, а также опциональное поле `order_product_type: str | None = None`.
- Поддержка сериализации: `model_config = ConfigDict(from_attributes=True)`.

### 3. Эндпоинты API (`backend/main.py`)
- `POST /api/tasks` — создание задачи.
  - Валидирует существование `order_id`, если он передан (иначе выбрасывает 404).
- `GET /api/tasks` — получение списка задач.
  - Доступно для роли `DIRECTOR` (ограничение роли).
  - Поддерживает фильтрацию через Query-параметры:
    - `order_id: int | None = None`
    - `status: TaskStatus | None = None`
- `PATCH /api/tasks/{id}/status` — быстрое обновление статуса.
  - Находит задачу по `id` (или 404).
  - Обновляет поле `status` и сохраняет изменения.
- `DELETE /api/tasks/{id}` — удаление задачи.
  - Доступно для роли `DIRECTOR`.
  - Удаляет задачу из БД.

### 4. Сидирование базы данных (`backend/seed_tasks.py`)
- Написание скрипта, который подключается к БД через `SessionLocal`.
- Очищает таблицу `tasks` (`db.query(Task).delete()`).
- Получает существующие заказы (например, "Кухня из массива дуба" Анны Смирновой, "Лестница деревянная на металлокаркасе" ИП Сидорова).
- Создает 3-4 тестовые задачи:
  1. *«Закупить дуб для Анны»* (привязана к заказу Анны, приоритет: `высокий`, статус: `нужно сделать`).
  2. *«Выезд на замер к Сидорову»* (привязана к заказу Сидорова, приоритет: `критический`, статус: `в процессе`, дедлайн на ближайшие дни).
  3. *«Подготовить чертежи»* (без привязки к конкретному заказу, приоритет: `средний`, статус: `нужно сделать`).
  4. *«Сделать уборку в мастерской»* (без привязки к заказу, приоритет: `низкий`, статус: `выполнено`).

### 5. Автоматизированные тесты (`backend/test_slice4.py`)
- Реализация тестов на чистом Python с использованием стандартного модуля `urllib.request`.
- Сценарии тестирования:
  - Успешное создание задачи с привязкой к заказу и без.
  - Попытка создания задачи с несуществующим `order_id` (ожидается `404` / `422`).
  - Получение списка задач и проверка фильтрации по `status` и `order_id`.
  - Обновление статуса задачи (PATCH).
  - Удаление задачи (DELETE).
  - Проверка ограничений доступа (возврат `403` при неверной роли для защищенных методов).

### 6. Визуальный интерфейс (`frontend/index.html`)
- **HTML/CSS дизайн**:
  - Создание нового раздела «Модуль Задач / Планировщик» под секцией заказов.
  - Использование гармоничной и премиальной палитры цветов для приоритетов:
    - `низкий`: светлый серо-синий фон (`#f1f2f6`), темный текст.
    - `средний`: мягкий оранжевый/желтый (`#ffe082`), темный текст.
    - `высокий`: насыщенный оранжево-красный (`#ff8f00`), светлый текст.
    - `критический`: глубокий красный (`#d32f2f`), светлый текст.
  - Таблица/Список задач с возможностью:
    - Отметить задачу как выполненную в один клик через Checkbox.
    - Изменить статус задачи через dropdown (`<select>`).
  - Форма быстрого создания задачи:
    - Поля: Название, Описание, Дедлайн (`due_date`), Приоритет, Привязка к заказу (динамически наполняемый выпадающий список `select` с заказами компании).
- **Виджет «Календарь / План на месяц»**:
  - Размещение под списком задач в виде структурированной хронологической ленты событий (Timeline).
  - Агрегация дат из нескольких источников:
    - Дедлайны задач (`due_date`) -> маркируются бейджем **"Задача"** (синий/фиолетовый цвет).
    - Даты доставки заказов (`delivery_date`) -> маркируются бейджем **"Доставка"** (зеленый цвет).
    - Даты монтажа заказов (`installation_date`) -> маркируются бейджем **"Монтаж"** (оранжевый цвет).
  - Сортировка событий по возрастанию даты с группировкой по дням для удобства планирования.
  - Подсветка просроченных событий (если дедлайн задачи или доставки прошел, а статус не завершен).

## Ожидаемый результат
Пользователь видит все текущие производственные и административные задачи в одном месте. Календарь наглядно отображает загрузку по дням (доставки, монтажи, дедлайны задач), что позволяет эффективно планировать работу столярного производства.
