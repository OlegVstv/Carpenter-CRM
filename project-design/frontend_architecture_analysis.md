# Анализ архитектуры фронтенда Carpenter CRM (Текущее состояние)

## Описание
В данном документе представлен детальный анализ текущей архитектуры фронтенда веб-приложения Carpenter CRM на момент реализации Среза №6 (Финансовый модуль).

---

## Текущее состояние архитектуры

На данный момент весь фронтенд CRM-системы представляет собой **монолитный одностраничный файл (Single Page App)**, расположенный в [index.html](file:///c:/My-Code-Factory/Antigravity/Carpenter-CRM/frontend/index.html).

### 1. Структура файла
Файл объединяет в себе все три слоя веб-страницы:
*   **HTML-разметка:** каркас интерфейса. Все разделы приложения (Срез 0 — Инфраструктура, Срез 1 — Лиды, Срез 2 — Клиенты, Срез 3 — Заказы, Срез 4 — Задачи, Срез 5 — Снабжение, Срез 6 — Финансы) расположены последовательно друг за другом в виде блоков (`.leads-table-container`).
*   **Стилизация (CSS в теге `<style>`):** ванильный CSS. Задает общую сетку приложения (в основном Grid/Flexbox для адаптивности), стили таблиц, форм ввода, статусов-бейджей и градиентных карточек (`.fin-card`).
*   **Логика приложения (JS в теге `<script>`):** чистый ванильный JavaScript (ES6).

### 2. Управление состоянием (State Management)
*   Данные хранятся во временной оперативной памяти на клиенте в глобальных переменных объекта `window` (например, `window.allClients`, `window.allOrders`, `window.allTasks`).
*   Синхронизация состояния происходит вручную: после успешного отправления мутирующего запроса (`POST`, `PATCH`, `DELETE`) вызывается цепочка функций обновления (например, `fetchClients()`, `fetchOrders()`), которая полностью перезапрашивает данные с бэкенда.

### 3. Взаимодействие с API (Data Fetching)
*   Используется нативный JavaScript `fetch` API.
*   Каждый модуль выполняет асинхронные HTTP-запросы к соответствующим эндпоинтам FastAPI в [main.py](file:///c:/My-Code-Factory/Antigravity/Carpenter-CRM/backend/main.py) (например, `/api/leads`, `/api/orders`, `/api/financials/summary`).
*   Реализована базовая обработка ошибок с выводом статуса операции пользователю через функции-хелперы уведомлений (`setActionStatus`, `setOrderActionStatus` и др.).

### 4. Рендеринг и DOM-манипуляции
*   Шаблонизация выполнена на строковых шаблонах ES6 (Template Literals). 
*   Обновление интерфейса происходит через перезапись свойства `innerHTML` целевых контейнеров (например, `tbody.innerHTML = '...'`).
*   Реализованы динамические элементы:
    *   **Inline-добавление строк** (`addNewLeadRow()`, `convertLeadRow()`), когда в таблицу временно вставляется строка с полями ввода (`<input>`, `<select>`).
    *   **Интерактивное изменение данных** (например, изменение статуса заказа/задачи/закупки прямо в таблице) через выпадающие списки `<select>` с мгновенной отправкой `PATCH` на бэкенд.

### 5. Ролевая модель на фронтенде (Auth/Role Stub)
*   Фронтенд запрашивает статус сервера через `/api/health` и считывает `current_role`.
*   Некоторые деструктивные действия (например, удаление платежа или закупки) визуально скрыты или заблокированы на основе роли пользователя (в рамках MVP используется заглушка `DIRECTOR`).

## Преимущества текущей архитектуры (почему это работает)

1.  **Мгновенный запуск и легкость:** Нет стадии сборки (Webpack/Vite), нет каталога `node_modules` на фронтенде. Страница загружается моментально.
2.  **Простота отладки:** Любое изменение кода сразу отображается в браузере по нажатию F5.
3.  **Отсутствие внешних зависимостей:** Минимальный риск поломки сборщика пакетов или несовместимости библиотек.

---

## Недостатки и архитектурные ограничения

1.  **Большой размер файла:** Файл превышает **2600 строк**. Читать, поддерживать и расширять код в таком формате сложно: переплетены разметка, стили и логика.
2.  **Неэффективный рендеринг:** Перезапись всего `innerHTML` при изменении одной записи заставляет браузер заново перерисовывать всю таблицу. Это сбрасывает фокус ввода, выделение текста и прокрутку.
3.  **Дублирование кода:** Функции отправки запросов, обработки ошибок и отображения уведомлений (`setActionStatus`, `setClientActionStatus`, `setOrderActionStatus`) дублируются для каждого модуля.
4.  **Сложность навигации:** По мере добавления срезов страница стала очень длинной (бесконечный скролл вниз). Пользователю неудобно видеть все таблицы одновременно.
5.  **Отсутствие типизации:** Изменения в API бэкенда (например, переименование поля в схемах Pydantic) легко пропустить на фронтенде, так как нет статической проверки типов.

## Рекомендации по развитию фронтенда

1.  **Внедрение модульной навигации (Sidebar / Tabs):** Скрыть бесконечную простыню таблиц. Сделать стильную левую панель навигации (Лиды, Клиенты, Заказы, Снабжение, Финансы, Настройки).
2.  **Разделение логики:** Вынести CSS в отдельный файл `index.css`, а JS-скрипты разбить по модулям (например, `leads.js`, `orders.js`, `finance.js`, `api.js` для общих запросов) и подключать их как ES-модули (`<script type="module">`).
3.  **Компонентный подход для UI-форм:** Заменить добавление строк в таблицы (`addNewLeadRow`) на аккуратные модальные окна (Modal Windows / Slide-over panels) с валидацией форм.
4.  **Улучшение UX:** Добавить скелетоны (skeletons) во время загрузки данных, тосты (toast-уведомления) вместо статичных текстовых меток статуса и улучшить интерактивность таблиц.
