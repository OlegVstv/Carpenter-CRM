## Часть 1: Инфраструктура Google Antigravity IDE (Базовый уровень)

### 1. Подключение GitHub (через MCP)
Для того чтобы ИИ-агент мог сам сохранять ваш код в репозиторий, выполните следующие шаги:
* **Создание ключа:** В GitHub перейдите в `Developer Settings` -> `Personal Access Tokens` -> `Tokens (classic)`. Создайте токен с правами `repo` и `workflow`.
* **Активация в IDE:** В панели Antigravity нажмите `...` -> `MCP Servers` -> `GitHub` -> `Connect` и вставьте ваш токен.
* **Результат:** Теперь вы можете сказать агенту: *"Создай новую ветку и отправь туда изменения по модулю Лидов"*, и он сделает это сам.

### 2. Запуск Агента-разработчика (Planning Mode)
Для реализации CRM по методике **Vertical Slices** крайне важно использовать режим планирования, чтобы избежать архитектурных ошибок:
1. **Переключение режима:** В панели агента выберите **Planning** (вместо Fast).
2. **Подача промпта:** Опишите задачу (например, *"Реализуй Модуль 2: создание Лида"*).
3. **Утверждение плана:** Агент создаст **Implementation Plan** (список действий). Проверьте его.
4. **Исполнение:** Нажмите `Approve Plan`. Агент сам создаст файлы, установит библиотеки и проверит работу через встроенный браузер.

### 3. Работа в среде Windows (Стабильное выполнение)
Чтобы терминал Antigravity IDE работал корректно и агент не "зависал" в ожидании ответа оболочки, всегда используйте команды с флагом завершения:

* **Оболочка:** Используйте префикс `powershell -c` (вместо pwsh).
* **Python/FastAPI:** Создание окружения: `powershell -c "python -m venv .venv"`.
* **Установка пакетов:** Всегда обращайтесь напрямую через путь к интерпретатору внутри проекта: 
  `powershell -c ".\.venv\Scripts\python.exe -m pip install [название]"`
* **Запуск сервера:** `powershell -c ".\.venv\Scripts\python.exe -m uvicorn main:app --reload"`

### 4. Работа с контекстом проекта в Antigravity (не в Gems, Gems должен всегда смотрит в Knowledge)
- **Источник истины:** Все технические и бизнес-требования хранятся в папке `/docs`. 
- **Приоритет:** Файлы в `/docs` имеют приоритет над встроенными знаниями ИИ. 
- **Команда агенту:** "Перед началом работы изучи регламенты в папке `/docs`".

## Часть 2: AI-Ready Layer (Direct Vertex AI SDK)

### 1. Концепция "Локального RAG"
- **Инструмент:** Google Cloud AI Platform (Vertex AI SDK).
- **Хранилище векторов:** pgvector в PostgreSQL.

### 2. Установка (Адаптация под Windows)
Если стандартная установка Genkit недоступна, агент использует прямой путь:
1. `powershell -c ".venv\Scripts\python.exe -m pip install google-cloud-aiplatform"`.

### 3. Реализация AI-логики (ai_config.py)
Агент должен инициализировать модель напрямую через `vertexai.generative_models.GenerativeModel`.

### 4. Промпт-инжиниринг (ОБЯЗАТЕЛЬНО)
Все вызовы модели должны содержать `system_instruction`:
*"Ты — эксперт-технолог столярного производства. Твои ответы должны базироваться только на загруженных регламентах. Если информации нет в базе — предложи позвать Начальника цеха"*.

## Часть 3: Работа с базой данных

### 1. Доступ к БД через терминал
Для проверки таблиц или расширения pgvector используйте:
`powershell -c "docker exec -it carpenter_crm_db psql -U admin -d carpenter_db"`

### 2. SQLAlchemy Standard
- Все модели данных (Лиды, Заказы) должны наследоваться от `Base` в `backend/database.py`.
- Использовать асинхронный драйвер для FastAPI не обязательно на этапе MVP, допустим `psycopg2`.

## Часть 4: Правила вертикальных срезов

### 1. Заглушки (Stubs)
- На Срезах №0-№3 авторизация заменяется "заглушкой" роли Директора (`CURRENT_ROLE=DIRECTOR` в `.env.local`).
- Не проектировать систему логина, пока не будет завершен Модуль 3 (Заказы).

### 2. Объем среза
Один срез = 1 таблица в БД + 1 API эндпоинт + 1 экран UI.

## Часть 5: Стратегия Фронтенда (Временный регламент)

### 1. Заморозка Node.js / Next.js
- **Статус:** Next.js заморожен до Среза №3 для снижения инфраструктурного давления.
- **Запрет:** Агенту запрещено использовать команды `npx`, `npm`, `node` на Срезах №0, №1 и №2.

### 2. Временный UI (Swagger / REST Client)
- Для проверки API-методов, создания и вывода Лидов используется автогенерируемая документация FastAPI Swagger UI (`http://localhost:8000/docs`).
- При необходимости вывода графических экранов для Директора на Срезах №1 и №2 используются нативные HTML+JS файлы, отдаваемые бэкендом (без сборщиков пакетов).
