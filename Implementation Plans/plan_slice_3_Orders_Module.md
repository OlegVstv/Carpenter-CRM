# План реализации Среза №3: Модуль Заказов (Orders Module)

## Описание
Данный срез посвящен реализации Модуля 3 «Проекты / Заказы клиентов» согласно `docs/business_logic.md`. 
Мы добавляем функционал управления заказами с привязкой к существующим клиентам, отслеживанием статусов и базовыми финансовыми и временными метриками.

## Шаги реализации

### 1. Слой БД (`backend/models.py`)
- **Добавление перечисления статусов (Enum)**:
  - Создать `OrderStatus` с 12 статусами: `принят`, `ожидает оплаты`, `согласован`, `передан в производство`, `ожидает материалы`, `в производстве`, `готов`, `на доставке`, `сдан клиенту`, `закрыт`, `приостановлен`, `проблемный`.
- **Создание модели `Order`**:
  - `id` (Integer, Primary Key, index).
  - `client_id` (Integer, ForeignKey к `clients.id`, NOT NULL).
  - `product_type` (String, тип изделия/работ).
  - `technical_spec` (String/Text, ТЗ/Описание).
  - `price` (Float, стоимость).
  - `paid_amount` (Float, оплачено, по умолчанию 0.0).
  - `status` (Enum `OrderStatus`, по умолчанию `принят`).
  - `created_at` (DateTime, timezone=True, default=func.now()).
  - `delivery_date` (Date, nullable).
  - `installation_date` (Date, nullable).
- **Связи (Relationships)**:
  - В модели `Order`: `client = relationship("Client", back_populates="orders")`.
  - В модели `Client`: `orders = relationship("Order", back_populates="client", cascade="all, delete-orphan")`.

### 2. Схемы API (`backend/schemas.py`)
- **`OrderBase`**: Базовые поля (`product_type`, `technical_spec`, `price`, `status`, `delivery_date`, `installation_date`).
- **`OrderCreate`**: Наследуется от `OrderBase`, добавляется `client_id`. `status` опционален, по умолчанию `принят`.
- **`OrderStatusUpdate`**: Поле `status` (тип `OrderStatus`).
- **`OrderResponse`**: Наследуется от `OrderBase`, содержит `id`, `client_id`, `paid_amount`, `created_at`. Конфиг `orm_mode = True` (или `from_attributes = True` для Pydantic v2).

### 3. Эндпоинты (`backend/main.py`)
Реализация эндпоинтов с роутером `/api/orders`:
- `POST /api/orders`: Создание заказа с валидацией существования `client_id`.
- `GET /api/orders`: Получение списка всех заказов (защищено роль-гардом `DIRECTOR`). Будет возвращать заказы вместе с базовой информацией о клиенте (опционально, для удобства фронтенда).
- `PATCH /api/orders/{id}/status`: Обновление статуса заказа (принимает `OrderStatusUpdate`).
- `DELETE /api/orders/{id}`: Удаление заказа.

### 4. Сидирование (`backend/seed_orders.py`)
- Написание скрипта `seed_orders.py`.
- Подключение к БД и создание 2-3 тестовых заказов, привязанных к уже существующим клиентам из сидов Среза №2.
- Демонстрация разных статусов и частичной оплаты.

### 5. Тестирование (`backend/test_slice3.py`)
- Написание тестов на чистом Python с использованием библиотеки `urllib`.
- Сценарии:
  1. Создание заказа (POST).
  2. Получение списка заказов (GET) и проверка прав (например, под ролью DIRECTOR).
  3. Изменение статуса заказа (PATCH).
  4. Удаление заказа (DELETE).

### 6. Интерфейс (`frontend/index.html`)
- **HTML-структура**: 
  - Добавление секции "Модуль Заказов" под таблицей клиентов.
  - Форма/Модальное окно для создания заказа (выбор клиента из списка (select), ввод типа изделия, ТЗ, цены, дат).
  - Таблица для отображения заказов.
- **CSS-стили (встроенные)**:
  - Наглядная цветовая индикация для 12 статусов заказа (например, зеленый для "готов" и "сдан", желтый для "в производстве", красный для "проблемный" и т.д.).
  - Стили для выпадающих списков прямо в таблице.
- **JavaScript-логика**:
  - `loadOrders()`: Загрузка списка заказов с бэкенда и их рендер в таблицу.
  - `createOrder()`: Сбор данных с формы и отправка POST запроса.
  - `updateOrderStatus(orderId, newStatus)`: Отправка PATCH запроса при изменении статуса в `<select>` внутри строки таблицы.
  - `deleteOrder(orderId)`: Удаление заказа (с подтверждением).
  - Заполнение `select` для выбора клиента при создании заказа (данные брать из существующего списка клиентов).

## Ожидаемый результат
Полноценный модуль работы с заказами, интегрированный с существующим модулем клиентов, работающий как на уровне БД и API, так и в визуальном интерфейсе с возможностью легкого управления статусами.
