# План реализации Среза №6: Финансовый модуль (Financial Light Module)

## Описание
Этот срез посвящен реализации Модуля 7 «Финансовый light-блок» согласно `docs/business_logic.md`. 
Мы внедряем систему отслеживания платежей по заказам клиентов (поступления средств, предоплаты, долги) и сопоставления их с расходами на снабжение (закупки под проекты).

---

## Шаги реализации

### 1. Слой БД (`backend/models.py`)

- **Модель `Payment` (Платеж / Факт поступления)**:
  - `id` (Integer, Primary Key, autoincrement).
  - `order_id` (Integer, ForeignKey к `orders.id`, NOT NULL).
  - `amount` (Float, сумма платежа, NOT NULL).
  - `payment_date` (Date, дата получения платежа, NOT NULL, по умолчанию текущая дата).
  - `comment` (String, комментарий к платежу, например: "Предоплата 50%", "Наличные", "Р/С ООО", nullable).
  - `created_at` (DateTime, по умолчанию UTC).
  
- **Связи (Relationships)**:
  - Связь `Order.payments` (один заказ может содержать много платежей, `cascade="all, delete-orphan"`, `back_populates="order"`).
  - Связь `Payment.order` (`back_populates="payments"`).

- **Статус оплаты заказа (Динамический расчет)**:
  - Статус оплаты (`payment_status`) и остаток долга (`remaining_balance`) будут вычисляться на бэкенде динамически для исключения десинхронизации данных:
    - `remaining_balance` = `price` - `paid_amount`
    - `payment_status`:
      - `"ожидает оплаты"`, если `paid_amount == 0`
      - `"предоплата"`, если `0 < paid_amount < price`
      - `"оплачен полностью"`, если `paid_amount >= price`

### 2. Схемы API (`backend/schemas.py`)

- **Платежи**:
  - `PaymentCreate`: поля `order_id`, `amount`, `payment_date` (nullable Date, по умолчанию сегодня), `comment` (nullable string).
  - `PaymentResponse`: наследует `PaymentCreate`, добавляет `id`, `created_at`, `payment_date` (гарантированная дата).
  - Настройка `ConfigDict(from_attributes=True)`.

- **Обновление `OrderResponse`**:
  - Добавить поля:
    - `remaining_balance: float` (вычисляется как `price - paid_amount`).
    - `payment_status: str` (вычисляется динамически).
    - `payments: List[PaymentResponse]` (опционально, список всех платежей по заказу).

- **Агрегированная сводка `FinancialSummaryResponse`**:
  - `total_revenue` (Float, сумма всех цен заказов).
  - `total_paid` (Float, общая сумма всех полученных платежей по заказам).
  - `total_debt` (Float, общая задолженность клиентов: `total_revenue - total_paid`).
  - `total_expenses` (Float, общая сумма всех не отмененных закупок материалов).
  - `net_profit` (Float, ожидаемая чистая прибыль: `total_revenue - total_expenses`).

---

### 3. Эндпоинты API (`backend/main.py`)

- **`POST /api/payments`** (Создание платежа):
  - Проверяет существование заказа по `order_id`.
  - Создает и сохраняет объект `Payment`.
  - Обновляет (увеличивает) поле `paid_amount` у связанного заказа `Order` на величину `amount`.
  - Фиксирует изменения в БД (`db.commit()`).
  - Возвращает созданный платеж.

- **`DELETE /api/payments/{payment_id}`** (Удаление платежа, роль-гард `DIRECTOR`):
  - Находит платеж в БД.
  - Вычитает его сумму из `paid_amount` соответствующего заказа `Order`.
  - Удаляет объект `Payment` из базы.
  - Фиксирует изменения (`db.commit()`).
  - Возвращает статус `{"status": "ok"}`.

- **`GET /api/financials/summary`** (Сводный финансовый дашборд):
  - Запрашивает все заказы для вычисления:
    - `total_revenue = sum(Order.price)`
    - `total_paid = sum(Order.paid_amount)`
    - `total_debt = total_revenue - total_paid`
  - Запрашивает все `SupplyRequest`, не имеющие статус `CANCELLED`, для вычисления:
    - `total_expenses = sum(SupplyRequest.quantity * SupplyRequest.actual_price)`
  - Вычисляет `net_profit = total_revenue - total_expenses`.
  - Возвращает JSON объект в соответствии с `FinancialSummaryResponse`.

### 5. Интеграционные тесты (`backend/test_slice6.py`)

Скрипт проверяет сквозной процесс на чистой БД с помощью библиотеки `urllib`:
1. Создание тестового клиента (`POST /api/clients`).
2. Создание заказа с ценой `100000` (`POST /api/orders`).
3. Проверка, что `paid_amount == 0` и `payment_status == "ожидает оплаты"`.
4. Внесение частичной оплаты на `40000` (`POST /api/payments`). Проверка, что статус стал `"предоплата"`.
5. Внесение оставшейся оплаты на `60000` (`POST /api/payments`). Проверка, что статус стал `"оплачен полностью"`.
6. Запрос финансовой сводки (`GET /api/financials/summary`), сверка полей `total_revenue`, `total_paid` и `total_debt`.
7. Удаление одного из платежей (`DELETE /api/payments/{id}`). Сверка возврата статуса заказа к `"предоплата"` и уменьшения `paid_amount`.

### 6. Интерфейс (`frontend/index.html`)

Добавляем новый премиальный раздел **"Финансы столярного производства"** в самый низ страницы:

- **Визуальный дашборд (Premium Financial Widgets)**:
  - Сетка из 5 карточек с яркими градиентами (Tailored HSL/CSS):
    - **Общая выручка**: Deep Slate Blue/Dark Gray `linear-gradient(135deg, #2c3e50, #34495e)`
    - **Получено средств**: Emerald Green `linear-gradient(135deg, #11998e, #38ef7d)`
    - **Остаток к получению (Долг)**: Coral/Red-orange `linear-gradient(135deg, #ff9966, #ff5e62)`
    - **Затраты на материалы**: Royal Purple `linear-gradient(135deg, #654ea3, #eaafc8)`
    - **Ожидаемая чистая прибыль**: Vibrant Teal `linear-gradient(135deg, #1f4037, #99f2c8)`
  - Карточки имеют тени и плавные микро-анимации при наведении (`transform: translateY(-4px)`).

- **Таблица финансового учета по заказам (Доходы)**:
  - Колонки: Номер заказа (ID), Клиент, Изделие, Стоимость, Оплачено, Остаток (Долг), Статус оплаты.
  - Цветовые бейджи для статуса оплаты:
    - `"ожидает оплаты"` — светло-красный фон, красный текст.
    - `"предоплата"` — светло-голубой фон, синий текст.
    - `"оплачен полностью"` — светло-зеленый фон, зеленый текст.
  - Интерактивные элементы:
    - Кнопка **"Внести платеж"** в строке заказа. При клике прямо под строкой заказа или в модальном окне открывается форма (Сумма, Комментарий, Дата).
    - Ссылка **"История платежей"**, показывающая список платежей с возможностью удаления (с роль-гардом `DIRECTOR`).

- **Таблица затрат и согласований по закупкам (Расходы)**:
  - Отображает закупки материалов со столбцами: Заказ, Материал, Поставщик, Сумма закупки, Статус закупки, Согласование (Approved / Draft).
  - Дает возможность директору в один клик изменить статус закупки с "черновик" на "согласовано" (кнопка **"Согласовать"** прямо из этой таблицы).

- **Скрипты JS**:
  - `fetchFinancials()`: делает запрос к `/api/financials/summary`, обновляет виджеты дашборда.
  - `loadFinancialOrders()`: загружает заказы, вычисляет долги и отрисовывает таблицу доходов.
  - `submitPayment(orderId)`: отправляет POST на `/api/payments`.
  - `deletePayment(paymentId)`: отправляет DELETE на `/api/payments/{id}`.
  - `approvePurchaseFromFinance(reqId)`: быстрое согласование заявки на закупку (PATCH `/api/supply-requests/{id}/status` со статусом `"согласовано"`).
